# Kaizen 2026-03-27

## 1. Manejo de errores en bloques catch

### Descripción
Basado en `docs/audits/AUDITORIA_2026_03_27.md`, el código está registrando o utilizando el objeto de error inferido como `unknown` en bloques `catch` de forma directa (`console.error(err)` o `console.error(error)`), sin extraer el mensaje utilizando un type guard (`instanceof Error`), lo cual incumple la directiva estricta de TypeScript.

### Ubicaciones
- `src/app/dashboard/page.tsx:57`
- `src/app/login/page.tsx:57`
- `src/app/companies/page.tsx:35`
- `src/app/companies/new/page.tsx:31`
- `src/app/companies/[id]/edit/page.tsx:29`
- `src/components/shared/DestructiveActionConfirm.tsx:56`
- `src/components/companies/company-form.tsx:113`
- `src/lib/api/admin-api-server.ts:36`
- `src/lib/config.ts:94`
- `src/components/ui/overlay-fix.tsx:59`

### Acción requerida
Reemplazar el uso directo de `err` o `error` en los bloques `catch` introduciendo un type guard seguro:
`const message = error instanceof Error ? error.message : String(error);`
Y reemplazar el log a `console.error(message)` o lo que corresponda según la ubicación.

## 2. Accesibilidad del componente Input compartido

### Descripción
El componente base de Input en `src/components/shared/Input.tsx` no soporta actualmente el paso de estados de error de forma explícita que modifiquen la accesibilidad del mismo mediante atributos `aria-invalid` y `aria-describedby`.

### Acción a realizar
- Añadir a la interfaz `InputProps` la posibilidad de recibir `hasError` (booleano) opcional, así como `errorMessageId` (string) opcional.
- Si `hasError` es verdadero, renderizar `aria-invalid="true"`.
- Si `errorMessageId` existe, renderizar `aria-describedby={errorMessageId}`.
- Reflejar estos cambios en los tests unitarios (`Input.spec.tsx`).
